# v0.2 – Stufenplan: PDF/E-Rechnungs-Import (Eingangsrechnungen)

Grundlage: `docs/02-kassenbuch.md`, Abschnitt „PDF/E-Rechnungs-Import (Kategorie 2)"

---

## Architektur-Entscheidung: Beleg-Speicherung

**Filesystem + eigene `belege`-Tabelle** (nicht SQLite-BLOB, nicht `beleg_pfad` direkt in `rechnungen`).

### Warum?

- SQLite-BLOBs für PDFs blähen die DB auf, GoBD-Export muss Dateien als echte Dateien liefern
- Eigene `belege`-Tabelle erlaubt Verknüpfung mit `rechnungen` UND `journal` (für spätere Direktbuchungen ohne Rechnung)
- SHA256-Hash in DB = GoBD-Nachweis der Unveränderbarkeit
- Einfache Erweiterung auf mehrere Belege pro Rechnung (Zwischentabelle) wenn nötig

### Datenmodell

```
belege
  id              INTEGER PRIMARY KEY
  dateiname       VARCHAR(500)   -- relativer Pfad: belege/YYYY/MM/name_uuid.pdf
  original_name   VARCHAR(255)   -- Ursprünglicher Dateiname vom Upload
  mime_type       VARCHAR(100)
  dateigroesse    INTEGER        -- Bytes
  sha256          VARCHAR(64)    -- SHA256 des Dateiinhalts (GoBD: Unveränderlichkeit)
  hochgeladen_am  TIMESTAMP

rechnungen.beleg_id  → belege.id   (nullable FK, Migration 23)
journal.beleg_id     → belege.id   (nullable FK, Migration 23, für spätere Direktbuchungen)
```

### Dateipfad-Strategie

```
~/.local/share/RechnungsFee/uploads/
  belege/
    2025/
      12/
        Rechnung-Mustermann_a3f5b8c2.pdf   ← stem(50 Zeichen) + "_" + uuid8 + suffix
```

### GoBD-Export-Erweiterung (Stufe 1 vorbereiten, Stufe 5 umsetzen)

Das bestehende GoBD-ZIP wird um folgende Dateien erweitert:
- `belege.csv` – Index: beleg_id, original_name, sha256, rechnung_id, journal_id, hochgeladen_am
- `belege/` – Ordner mit den tatsächlichen Dateien

## Stufe 1 – Beleg anhängen *(keine Analyse, sofort nützlich)* ✅ v0.2.3

**Ziel:** Jede Eingangsrechnung bekommt einen Datei-Anhang. Keine Auswertung, nur Upload/Download.

### Backend
- **Migration 23:** `belege`-Tabelle erstellen, `beleg_id` zu `rechnungen` und `journal` hinzufügen
- `POST /api/rechnungen/{id}/beleg` – multipart/form-data, speichert Datei + DB-Eintrag + SHA256
- `GET  /api/rechnungen/{id}/beleg` – liefert Datei als FileResponse
- `DELETE /api/rechnungen/{id}/beleg` – löscht Datei + `belege`-Eintrag + leert FK

### Frontend
- Im Eingangsrechnung-Detail-Panel: Upload-Button + Dateiname + Download-/Öffnen-Link
- Vorhandener Beleg: Dateiname, Dateigröße, Datum + Löschen-Button
- Kein Beleg: Upload-Dropzone (oder einfacher Dateiauswahl-Button)

### Neue Pakete
Keine – `python-multipart` ist durch FastAPI bereits vorhanden

### Akzeptierte Dateitypen
- `application/pdf`
- `image/jpeg`
- `image/png`
- `image/tiff`

## Stufe 2 – Format-Erkennung + ZUGFeRD/XRechnung Parsing *(Auto-Fill)* ✅ v0.2.4

**Ziel:** Hochgeladene Datei analysieren → Felder im Eingangsrechnungs-Formular vorausfüllen.

### Backend
- **Neue Pakete:** `factur-x`, `lxml`, `pypdf`
- `POST /api/rechnungen/analysieren` – nimmt Datei, gibt erkannte Felder zurück (keine DB-Änderung)
- Format-Erkennung:
  - `.xml` → XRechnung (`lxml`)
  - PDF mit eingebettetem XML → ZUGFeRD (`factur-x.get_facturx_xml_from_pdf`)
  - sonst → normales PDF (Stufe 4: OCR)
- Antwort-Schema:
  ```json
  {
    "format": "zugferd|xrechnung|pdf",
    "felder": {
      "externe_belegnr": "RE-2025-0042",
      "datum": "2025-12-01",
      "faellig_am": "2025-12-31",
      "gesamt_brutto": "119.00",
      "ust_satz": 19,
      "lieferant_name": "Muster GmbH",
      "lieferant_ust_id": "DE123456789",
      "positionen": [...]
    },
    "warnungen": ["BT-9: Fälligkeitsdatum fehlt"]
  }
  ```

### Frontend
- Neuer Einstieg auf RechnungenPage: Button „Rechnung importieren" ✅
- Upload → API-Call → Formular vorausgefüllt, alle Felder editierbar ✅
- Warnungen als gelbes Banner ✅
- Plain PDF: öffnet sich automatisch im Viewer beim Klick auf „Rechnung erstellen" ✅
- Originaldatei wird nach dem Speichern automatisch als Beleganhang angehängt ✅

## Stufe 3 – Validierung *(Errors blockieren, Warnings informieren)*

**Ziel:** Pflichtfelder prüfen bevor Import bestätigt wird.

### Backend
- `analysieren`-Endpoint um Pflichtfeld-Prüfung erweitern (BT-1, BT-2, BT-27, BT-112 etc.)
- Antwort: zusätzlich `"errors": [...]`
- Errors → Import gesperrt; Warnings → Import möglich mit Hinweis

### Frontend
- Rotes Banner + Liste wenn `errors` vorhanden → Import-Button deaktiviert
- Gelbes Banner bei nur `warnungen` → Import möglich

*Kein XML-Editor in dieser Stufe.*

## Stufe 4 – OCR für normale PDFs *(optional, nutzerkontrolliert)*

**Ziel:** Wenn kein strukturiertes XML gefunden → Felder per Regex/OCR extrahieren.

### Backend
- **Neue Pakete:** `pdfplumber` (maschinenlesbare PDFs), `pytesseract` + `Pillow` (Scans/Fotos)
- `analysieren` bei `format=pdf`: erst `pdfplumber.extract_text()`, dann Regex für:
  - Rechnungsnummer (`Rechnung(?:snummer)?[:\s#]+(\S+)`)
  - Datum (`\d{2}\.\d{2}\.\d{4}`)
  - Betrag (größter EUR-Betrag im Text)
  - Name aus ersten Textzeilen
- Konfidenz pro Feld: `"konfidenz": {"externe_belegnr": 0.9, "datum": 0.6}`

### Frontend
- OCR-Dialog vor der Analyse (bei normalem PDF):
  - Ja, Daten automatisch ausfüllen (empfohlen)
  - Nein, manuell eingeben
  - Auswahl merken
- Felder mit Konfidenz < 0.7 → gelb hinterlegt
- OCR-Einstellung in Stammdaten: immer fragen / immer / nie

## Stufe 5 – PDF/A-Archivierung *(GoBD-konform)*

**Ziel:** Normale PDFs beim Speichern zu PDF/A-3 konvertieren.

### Backend
- **Neue Pakete:** `ocrmypdf` (beinhaltet PDF/A-Konvertierung, braucht Ghostscript als Systempaket)
- **Migration 24:** `beleg_pdfa_pfad VARCHAR(500)` in `belege`
- Nach erfolgreichem Upload:
  - ZUGFeRD/XRechnung → bereits PDF/A-3, `beleg_pdfa_pfad = dateiname`
  - Normales PDF → `ocrmypdf --pdfa 3 --skip-text input.pdf output_pdfa.pdf`
  - Bild → erst zu PDF, dann ocrmypdf
- `GET /api/rechnungen/{id}/beleg?version=pdfa` für archivierten Stand
- GoBD-Export ergänzt: `belege.csv` + `belege/`-Ordner im ZIP

### Frontend
- Download-Button unterscheidet Original / PDF/A

## Aufwand-Schätzung

| Stufe | Aufwand | Mehrwert |
|-------|---------|----------|
| 1 – Upload | ~4 Std | Belege sofort anhängbar |
| 2 – Parsing | ~1 Tag | ZUGFeRD/XRechnung füllt Formular automatisch aus |
| 3 – Validierung | ~2 Std | Fehlerhafte E-Rechnungen werden erkannt |
| 4 – OCR | ~1 Tag | Normale PDF-Rechnungen halb-automatisch erfasst |
| 5 – PDF/A | ~4 Std | GoBD-konforme Langzeitarchivierung |
